> **Версия:** актуально на 2026-06-29. Документ обязан обновляться вместе с

## 1. Введение и место роли в иерархии

**Arena** — мультитенантная платформа продажи билетов (наследник Bil24/TixGear).
Платформа разделена на изолированные домены ответственности, и роль
`platform_superadmin` стоит на самом верху этой пирамиды.

### Иерархия ролей (упрощённо)

```
platform_superadmin   ← вы
        │
        ├── platform_operator        (внутренний back-office Arena/Bil24)
        │
        ├── operator_network         (бизнес-сущность: сеть операторов)
        │       │
        │       └── network_operator (управляет сетью)
        │               │
        │               ├── organizer (организатор мероприятий)
        │               │       └── agent (агент по продажам)
        │               │
        │               └── scanner  (контролёр на входе)
        │
        └── customer                 (B2C-клиент, не работает в админке)
```

> Подробный маппинг на legacy Bil24-роли — см. `docs/rbac/legacy_bil24_role_mapping.md`.

### Что делает суперадмин

- Заводит операторские сети, организации, ключевые учётные записи.
- Управляет глобальной конфигурацией: платёжные провайдеры, шаблоны,
  feature flags, ограничения сети.
- Расследует инциденты безопасности, использует cross-tenant read/write
  с обязательным `reason context`.
- Проводит аудит и ротацию секретов.
- **НЕ** занимается операционной продажей билетов — для этого есть
  `network_operator` и `organizer`.

### Чем суперадмин отличается от других ролей

| Признак | `platform_superadmin` | `platform_operator` | `network_operator` |
|---|---|---|---|
| Scope | вся платформа | вся платформа (без cross-tenant write) | одна сеть |
| Cross-tenant действия | **да** (с `reason`) | только чтение | нет |
| Управление сетями | **да** | нет | нет |
| Управление платёжными провайдерами | **да** | конфигурация | только привязка |
| Просмотр audit-log | **да, глобально** | глобально | в рамках сети |

### Область ответственности

Суперадмин **отвечает** за:
- безопасность платформы (ротация секретов, разбор инцидентов);
- корректную онбординг-цепочку новых клиентов-организаций;
- работоспособность интеграций (платежи, webhook, WordPress/Bil24);
- соответствие compliance-требованиям (PCI, GDPR/152-ФЗ).

Суперадмин **не отвечает** за:
- ежедневную операционную продажу билетов конкретной организации;
- содержание мероприятий, маркетинг, ценообразование.

## 2. Первый вход в систему

### Адрес админ-панели

- **Production:** `https://admin.arena.example.com`
- **Staging:** `https://admin.staging.arena.example.com`
- **Local dev:** `http://localhost:3001` (см. `apps/admin-web/README.md`)

### Шаги первого входа

1. **Получите учётку** от службы платформы (создаётся через миграцию
   `0034_superadmin.sql` или CLI `arena-migrate seed superadmin`).
2. **Откройте URL** админ-панели.
3. **Введите логин/пароль**, выданные при создании.
4. **MFA (обязательно для prod):** при первом входе включите TOTP-аутентификатор
   (Google Authenticator, 1Password, Authy). Сохраните backup-коды в защищённом
   хранилище.
5. **Смена временного пароля:** система потребует задать новый пароль не менее
   12 символов, с цифрой, заглавной буквой и спецсимволом.
6. **Выбор scope:** в правом верхнем углу — переключатель scope. По умолчанию
   стоит `Platform`. Можно выбрать конкретную `operator_network` или
   `organization`, чтобы видеть данные «глазами» этого тенанта.
7. **Заголовок `X-Admin-Reason` (cross-tenant):** для любых *записывающих*
   действий за пределами вашего собственного scope система потребует указать
   **причину** в диалоге (поле «Reason / Обоснование»). Эта причина уйдёт в
   `audit_log` и будет видна в отчётах compliance.

> ⚠️ **Никогда не передавайте** учётные данные суперадмина по e-mail,
> мессенджеру или другим небезопасным каналам. Для временного доступа
> используйте сессионные токены с ограниченным TTL.

## 3. Обзор интерфейса админ-панели

После входа вы увидите:

### Главное меню (левая колонка)

- **Dashboard** — глобальный обзор: количество активных сетей, организаций,
  мероприятий, объём продаж за сутки/неделю/месяц.
- **Networks** — операторские сети.
- **Organizations** — организации-клиенты.
- **Users** — пользователи и роли.
- **Venues** — площадки.
- **Events** — мероприятия.
- **Sales** — каналы продаж, лимиты, скидки.
- **Payments** — платёжные провайдеры и их конфигурации.
- **Scanners** — управление сканерами и контрольными точками.
- **Reports** — отчётность и выгрузки.
- **Reconciliation** — отчёты сверки (см. миграцию `0041_reconciliation_reports.sql`).
- **Webhooks** — доставка webhook'ов, ретраи.
- **Audit Log** — журнал действий.
- **System** — настройки платформы, feature flags, секреты.

### Верхняя панель

- **Глобальный поиск** (`Ctrl+K` / `Cmd+K`) — поиск по `id`, e-mail, названию
  организации, номеру билета.
- **Переключатель scope** — `Platform / Network / Organization`.
- **Нотификации** — критические уведомления: упавшие webhook'и, ошибки
  платежей, признаки brute-force.
- **Профиль** — настройки учётки, ключи API, выход.

### Глобальный дашборд

Виджеты по умолчанию:
- KPI: GMV, число заказов, средний чек, процент конверсии.
- Графики продаж за 24ч / 7д / 30д.
- Топ-5 организаций, топ-5 мероприятий.
- «Здоровье платформы»: задержка очередей, лаг outbox, количество failed
  webhook deliveries.

## 4. Управление операторскими сетями (operator networks)

Operator network — это **бизнес-сущность**, объединяющая нескольких
организаторов под управлением одного `network_operator`. Используется
для франшиз, билетных операторов, дистрибьюторских сетей.

### 4.1. Создать сеть

1. Меню → **Networks** → кнопка **«+ New network»**.
2. Заполните:
   - **Name** — публичное название сети.
   - **Slug** — машиночитаемый ID (lower-snake-case, уникально).
   - **Description** — внутреннее описание.
   - **Default currency** — основная валюта.
   - **Locale** — основной язык (ru/en/…).
   - **Contact email** — email ответственного `network_operator`.
3. Нажмите **Create**.

### 4.2. Назначить network_operator

1. Откройте созданную сеть → вкладка **«Users»** → **«+ Assign»**.
2. Выберите пользователя (или создайте нового, см. раздел 6).
3. Установите роль = `network_operator`.
4. При необходимости включите права `network.manage_agents`,
   `network.manage_organizations`, `network.view_reports`.
5. Сохраните. Пользователь получит уведомление на e-mail.

### 4.3. Привязать организации к сети

1. Вкладка **«Organizations»** в карточке сети → **«+ Attach»**.
2. Выберите существующую организацию из списка или создайте новую
   (раздел 5) сразу с привязкой к сети.
3. Подтвердите. Привязка фиксируется в `operator_network_organizations`.

### 4.4. Архивация / удаление сети

- **Архивация** (`Archive network`) — мягкое удаление. Сеть исчезает из
  основных списков, но остаётся в `audit_log` и отчётах. Действия по сети
  блокируются, привязанные организации помечаются как `detached`.
- **Полное удаление** — только через CLI `arena-migrate` и согласование с
  безопасностью. В UI **не реализовано** по соображениям compliance.

> ⚠️ Перед архивацией убедитесь, что у сети нет активных мероприятий с
> открытой продажей. Иначе продажи будут принудительно закрыты.

## 5. Управление организациями

Организация — отдельный тенант, владелец мероприятий и площадок.

### 5.1. Создать организацию

1. Меню → **Organizations** → **«+ New organization»**.
2. Заполните:
   - **Legal name** — официальное наименование.
   - **Display name** — отображаемое имя.
   - **Slug** — уникальный машиночитаемый ID.
   - **INN / KPP / OGRN** (для РФ) или **VAT ID** (EU).
   - **Address** — юридический и фактический.
   - **Contact** — основной контакт (имя, e-mail, телефон).
   - **Network** — привязка к operator network (опционально).
3. Сохраните. Будет создана запись в `organizations` с автогенерированным
   UUIDv7.

### 5.2. Заполнение реквизитов и документов

Вкладка **«Legal & Billing»** в карточке организации:
- Банковские реквизиты (счёт, БИК, корр. счёт).
- Документы: устав, выписка ЕГРЮЛ, договор оферты.
- Налоговый статус (ОСН/УСН/НДС).
- Подписант (ФИО, должность, основание).

Эти данные используются в:
- сервисном биллинге (раздел 12);
- генерации актов и счетов-фактур;
- webhook payload'ах для интеграций.

### 5.3. Архивация и восстановление

- **Archive** — soft-delete: организация скрывается, продажи блокируются.
- **Restore** — возвращает архивную организацию в активные. Все привязки
  (сеть, пользователи, площадки) сохраняются.

### 5.4. Перенос организации в другую сеть

1. Карточка организации → кнопка **«Move to another network»**.
2. Выберите целевую сеть.
3. **Обязательно укажите Reason** (cross-tenant действие).
4. Подтвердите. Перенос фиксируется в `audit_log` с событием
   `organization.network_changed`.

## 6. Управление пользователями и ролями

### 6.1. Создать пользователя

1. Меню → **Users** → **«+ New user»**.
2. Введите:
   - **Email** (уникальный, используется как логин).
   - **Display name**.
   - **Locale** (ru/en).
   - **Send invite** — если включено, на e-mail уйдёт ссылка для установки
     пароля.
3. Сохраните.

### 6.2. Назначение ролей

В карточке пользователя → вкладка **«Roles & Memberships»**:

| Роль | Scope | Когда назначать |
|---|---|---|
| `platform_superadmin` | Platform | Сотрудник Arena/Bil24 с полным доступом. |
| `platform_operator` | Platform | Внутренний back-office (контент, фронтенды, эквайринг). |
| `network_operator` | Operator network | Управляющий конкретной operator network. |
| `organizer` | Organization | Сотрудник организатора мероприятий. |
| `agent` | Organization | Агент по продажам (через `organizer` или `network`). |
| `scanner` | Venue/Event | Контролёр на входе. |
| `reporter` | scoped read | Read-only доступ к отчётам (через permissions). |

Маппинг на legacy Bil24-роли — см. `docs/rbac/legacy_bil24_role_mapping.md`.

### 6.3. Membership в организации

Один пользователь может иметь несколько membership'ов (в разных
организациях, с разными ролями). Назначаются в карточке организации →
вкладка **«Users»** или в карточке пользователя → **«Memberships»**.

### 6.4. Сброс пароля

1. Карточка пользователя → **«Reset password»**.
2. Подтвердите. На e-mail уйдёт ссылка с TTL = 60 минут.
3. Если пользователь не получил письмо, можно сгенерировать
   одноразовый код (показывается ОДИН РАЗ) и передать через
   защищённый канал.

### 6.5. Блокировка / разблокировка

- **Block user** — мгновенно отзывает все активные сессии и токены.
  Пользователь не может войти.
- **Unblock** — восстанавливает доступ.
- Блокировка фиксируется в `audit_log` с указанием `reason`.

> ⚠️ Блокировка `platform_superadmin` другим суперадмином требует
> подтверждения двух суперадминов (4-eyes principle, см. раздел 14).

## 7. Управление площадками (venues)

### 7.1. Создать площадку

1. Меню → **Venues** → **«+ New venue»**.
2. Заполните:
   - **Name** — публичное название.
   - **Organization** — владелец.
   - **Address** — страна, город, улица, индекс, координаты (lat/lng).
   - **Timezone** — обязательно (например `Europe/Moscow`).
   - **Capacity** — общая вместимость (для площадок без посекторной
     схемы).
3. Сохраните.

### 7.2. Схема зала, сектора, места

Вкладка **«Seating plan»**:
- **Upload SVG** — загрузить SVG-схему зала. Каждое место должно иметь
  `id` вида `sector-row-seat`.
- **Map sectors** — задать соответствие SVG-элементов → секторам и местам
  в базе.
- **Bulk import** — CSV: `sector,row,seat,x,y,price_category`.
- **Validate** — кнопка проверки на дубликаты и пустые ячейки.

Подробнее — см. директорию `06_venue_maps_and_seating/`.

### 7.3. Статусы площадки

- `draft` — черновик, не виден на фронте.
- `active` — доступна для создания мероприятий.
- `archived` — скрыта, активные мероприятия достоигрываются, новые
  создать нельзя.

### 7.4. Привязка к организации

При создании площадка автоматически привязывается к выбранной
организации. Перенос на другую организацию — отдельная операция с
обязательным `reason` и подтверждением (cross-tenant действие).

## 8. Создание и публикация мероприятия

> Обычно мероприятия создаёт `organizer`, но суперадмин может делать это
> в режиме «от имени» (impersonation), переключив scope.

### 8.1. Базовая информация

1. Меню → **Events** → **«+ New event»**.
2. Выберите:
   - **Organization** (владелец).
   - **Venue** (из площадок организации).
3. Заполните:
   - **Title** (мультиязычно: ru/en).
   - **Description** (markdown).
   - **Category** (концерт, спектакль, спорт, лекция…).
   - **Cover image** (1200×630, JPEG/PNG).
   - **Start at / End at** (с учётом timezone площадки).
4. Сохраните как `draft`.

### 8.2. Сессии (sessions)

Для повторяющихся мероприятий заведите несколько сессий:
- **Session date/time**
- **Doors open**
- **Duration**
- **Status** (`scheduled`, `cancelled`, `postponed`)

### 8.3. Категории билетов и цены

Вкладка **«Tickets»**:
- **Categories**: VIP, Партер, Балкон, и т.д. Привязываются к секторам схемы.
- **Prices**: для каждой категории — базовая цена, валюта, динамические
  правила (early bird, last minute).
- **Limits**: максимум билетов на одного покупателя, лимит per channel.

### 8.4. Публикация

1. Перевести статус в `published` — кнопка **«Publish»**.
2. Система проверит:
   - заполнены ли цены для всех мест/секторов;
   - настроен ли хотя бы один канал продаж;
   - указан ли платёжный провайдер;
   - не пересекается ли время с другим мероприятием на той же площадке.
3. После публикации мероприятие появляется в каналах продаж в течение
   1–2 минут (обновление кэша CDN).

## 9. Управление продажами и каналами продаж

### 9.1. Каналы продаж (sales channels)

Каналы — это «точки входа» к билетам: сайт организатора, виджет на
сторонних сайтах, WordPress-плагин, кассы, агентские интерфейсы.

Меню → **Sales** → **Channels**.

Для каждого канала:
- **Name / Slug**.
- **Type**: `web`, `widget`, `pos`, `agent`, `bil24_compat`, `wordpress`.
- **Organization** или **Network** (владелец).
- **API key** (генерируется один раз, видна один раз — после сохранения
  только маска `sk_live_••••XXXX`).
- **Allowed origins** (CORS).
- **Status** (`active`, `disabled`).

### 9.2. Маскирование секретов

После генерации API-ключа полное значение **больше никогда не отображается**.
Если ключ утерян — нужно создать новый и отозвать старый (`Revoke`).

### 9.3. Активация / деактивация канала

- **Activate** — канал начинает принимать заказы.
- **Deactivate** — канал блокируется. Существующие сессии checkout
  завершаются успешно (5 мин grace period), новые не создаются.
- **Revoke API key** — мгновенно отзывает ключ.

### 9.4. Фронтенд-каналы

Виджеты и WordPress-плагин дополнительно конфигурируются в разделе
**Frontends**: размеры, темы оформления, разрешённые поля заказа.

## 10. Платёжные системы

### 10.1. Конфигурация провайдера

Меню → **Payments** → **Providers** → **«+ New provider config»**.

Поддерживаемые провайдеры:
- Stripe
- YooKassa (бывший Яндекс.Касса)
- CloudPayments
- Tinkoff Acquiring
- PayPal (для международных)
- Bil24 legacy gateway (для совместимости)

Поля:
- **Provider** — выбор из списка.
- **Mode** — `test` / `live`.
- **Organization / Network** — кому принадлежит конфигурация.
- **Credentials** — API-key, secret, shop_id (зависит от провайдера).
  Секреты хранятся зашифрованными, в UI показывается только маска.
- **Webhook URL** — URL, который провайдер вызовет для уведомлений
  (автогенерируется: `https://api.arena…/v1/webhooks/payments/{provider}/{config_id}`).
- **Webhook secret** — для проверки подписи.

### 10.2. Тестовый и боевой режим

- В режиме `test` используются тестовые карты провайдера (4242…, 5555… и т.д.).
- Переход в `live` требует:
  - заполненных реквизитов организации (раздел 5.2);
  - подписанного договора с провайдером (вне платформы);
  - подтверждения от `platform_superadmin` с `reason`.

### 10.3. Проверка webhook'ов

1. Меню → **Payments** → **Webhooks**.
2. Фильтр по провайдеру и периоду.
3. Колонки: timestamp, event type, status (`delivered` / `failed`),
   attempts, last error.
4. Кнопка **«Replay»** — отправить webhook на обработку повторно.
5. Кнопка **«Send test»** — отправить тестовый webhook (нужен в случае
   первичной настройки).

## 11. Сканеры и контроль входа

### 11.1. Настройка интеграции сканеров

Меню → **Scanners** → **Devices** → **«+ Register device»**.

Поля:
- **Device name** (например: «Главный вход — терминал 1»).
- **Venue / Event** (привязка).
- **Hardware type** — `ios`, `android`, `dedicated_terminal`, `webcam`.
- **Activation code** — 6-значный код, вводится на устройстве для

### 11.2. Учётки сканировщиков

1. Меню → **Users** → создать пользователя с ролью `scanner`.
2. Привязать к событию или площадке.

### 11.3. Мониторинг проходов

В разделе **Scanners** → **Live**:
- Реал-тайм-лог сканирований.
- Цвета: зелёный — валид, красный — отказ (дубликат, неверный билет,
  отменён, не то событие).
- Счётчики посещаемости по входам.

### 11.4. Отчёт по проходам

После события — раздел **Reports** → **Attendance**:
- Процент посещаемости (отсканировано / продано).
- По воротам, по секторам, по типам билетов.

## 12. Отчётность и реконсиляция

### 12.1. Запуск отчётов

Меню → **Reports**:

| Отчёт | Описание |
|---|---|
| Sales summary | GMV, количество заказов, средний чек |
| Sales by event | Детально по мероприятиям |
| Refunds | Возвраты и причины |
| Channel performance | Конверсия по каналам |
| Tax | НДС, по ставкам |
| Custom | Произвольный SQL (только superadmin) |

Параметры: период, организация/сеть, валюта, формат (CSV/XLSX/JSON/PDF).
Большие отчёты запускаются в фоне; готовый файл придёт в нотификации.

### 12.2. Реконсиляция (reconciliation reports)

С момента миграции `0041_reconciliation_reports.sql` платформа автоматически
строит сверки между:
- продажами в Arena ↔ платёжным провайдером;
- продажами ↔ webhook'ами WordPress/Bil24;
- зарезервированными местами ↔ выпущенными билетами.

Меню → **Reconciliation**:
- **Daily** — сверка за прошлые сутки (запускается в 03:00 timezone платформы).
- **Custom range** — ручной запуск.
- **Discrepancies** — список расхождений, для каждого можно посмотреть
  raw payload и открыть инцидент.

### 12.3. Биллинг услуг платформы

Меню → **Billing**:
- Тарифные планы (transaction fee, monthly fee, hybrid).
- Привязка тарифа к организации/сети.
- Генерация инвойсов за период.
- Статусы (`draft` / `issued` / `paid` / `overdue`).
- Выгрузка в формате 1С или CSV для бухгалтерии.

## 13. Webhook delivery и интеграции (WordPress/Bil24)

### 13.1. Подключение интеграции

Меню → **Webhooks** → **Endpoints** → **«+ New endpoint»**.

Поля:
- **URL** — куда слать (например `https://wp.client.com/wp-json/bil24/webhook`).
- **Events** — список событий (order.created, order.paid, ticket.issued,
  ticket.scanned, refund.created…).
- **Auth** — `none`, `basic`, `hmac_sha256`, `bearer`.
- **Secret** — для HMAC.
- **Organization** / **Network** / **Platform** — owner.
- **Retry policy** — по умолчанию: 6 попыток, экспоненциальный backoff
  (1m → 5m → 30m → 2h → 12h → 24h).

### 13.2. Проверка доставки

Меню → **Webhooks** → **Deliveries**:
- Фильтр: endpoint, event, status.
- Для каждой доставки: request payload, response code, body, timing.
- **Replay** — повтор доставки.
- **Pause endpoint** — временно остановить отправку.

### 13.3. Совместимость с Bil24 (legacy gateway)

Для клиентов, мигрирующих с Bil24, доступен compatibility gateway,
который эмулирует старый XML/SOAP API. Конфигурация — в **Channels**
→ тип `bil24_compat`. Подробности — `08_architecture/12_master_platform_specification_ru.md`.

### 13.4. WordPress-плагин

- Плагин ставится через `wp-admin` (см. `apps/wordpress-plugin/README.md`).
- Регистрируется как канал типа `wordpress` (раздел 9.1).
- Webhook от Arena → WP отправляется на эндпоинт плагина с подписью HMAC.

## 14. Безопасность и аудит

### 14.1. Просмотр audit-log

Меню → **Audit Log**.

Каждая запись содержит:
- `timestamp` (UTC, миллисекунды);
- `actor` — пользователь / API-key;
- `action` — например `organization.created`, `user.role_changed`;
- `target` — затронутая сущность (тип + id);
- `scope` — platform / network / organization;
- `reason` — для cross-tenant действий;
- `request_id`, `correlation_id` — для трассировки;
- `ip_address`, `user_agent` — для security review.

Фильтры: по actor, по типу события, по scope, по периоду, по reason.

### 14.2. Расследование инцидентов

При срабатывании сигналов SIEM:
1. Зайдите в **Audit Log**, отфильтруйте по затронутому объекту.
2. Сопоставьте `correlation_id` с логами API (Loki/ELK).
3. Если найден компрометированный токен — отзовите его в **Users** →
   **Sessions** → **Revoke all**.
4. Запустите ротацию секретов (раздел 14.3).
5. Создайте incident в `00_project_control/`.

### 14.3. Ротация секретов

Меню → **System** → **Secrets**:
- **API keys** — список всех каналов, для каждого кнопка **Rotate**.
- **Webhook secrets** — ротация HMAC-секретов для исходящих webhook'ов.
- **JWT signing key** — ротация платформенного ключа (требует
  4-eyes-подтверждения и приведёт к разлогину всех пользователей).
- **DB encryption key** — только через CLI, не в UI.

Рекомендованная частота:
- API keys: при каждом инциденте или раз в 12 месяцев.
- Webhook secrets: при инциденте или по запросу клиента.
- JWT key: 1 раз в 6 месяцев, плановое окно обслуживания.

### 14.4. Hardening checklist

- [ ] MFA включён у всех `platform_superadmin` и `platform_operator`.
- [ ] Все API-keys используют HTTPS-only origins.
- [ ] Все webhook'и используют HMAC или mTLS.
- [ ] В `audit_log` нет действий с пустым `reason` при cross-tenant scope.
- [ ] Backup БД работает (см. `ops/`), последняя успешная копия < 24ч.
- [ ] Все секреты в `System → Secrets` имеют возраст < 12 месяцев.
- [ ] Отсутствуют orphan-membership (пользователь без активной организации).

Подробнее — `08_architecture/10_compliance_security_privacy_ru.md`.

## 15. Тестовая зона: prod vs test scope

### Что такое test scope

Платформа разворачивается в нескольких окружениях:
- **Production** (`prod`) — реальные деньги, реальные клиенты.
- **Staging** — preview новых релизов, тестовые карты провайдеров,
  изолированная БД.
- **Local** — разработческие копии.

В каждом окружении есть **seed-данные** для тестов:
- организации `Test Org #1..3`,
- площадки `Demo Venue`,
- пользователи с известными e-mail (`test+org1@arena.local`),
- платёжные конфигурации в режиме `test`.

### Как НЕ перепутать prod и test

1. **Цвет хедера**: prod — серый, staging — оранжевый, local — фиолетовый.
2. **Бэйдж в правом верхнем углу**: `PROD` / `STAGING` / `DEV`.
3. **Подтверждение для опасных действий**: в prod при удалении/архивации
   требуется ввести slug объекта.
4. **Разные домены** — никогда не работайте в prod-домене для проверки
   гипотез. Создайте всё на staging.
5. **Все seed-аккаунты** имеют префикс `test+` или домен `@arena.local` —
   проверяйте, что НЕ работаете под seed-учёткой в проде.

> ⚠️ Все действия в prod пишутся в `audit_log` без возможности удаления.
> Любое тестирование «на проде» оставляет неизгладимый след.

## 16. Типовые сценарии (cookbook)

### 16.1. Завести нового клиента-организатора с нуля

1. **Networks** → выбрать или создать сеть, в которой будет работать клиент.
2. **Organizations** → **+ New organization** → заполнить юр.данные.
3. Привязать организацию к сети (вкладка Network в карточке организации).
4. **Users** → создать пользователя с ролью `organizer` → пригласить по
   e-mail.
5. Membership: добавить пользователя в созданную организацию.
6. **Venues** → создать первую площадку, привязать к организации.
7. **Payments** → настроить хотя бы один payment provider (test mode).
8. **Sales** → создать минимум один канал продаж.
9. Передать клиенту:
   - URL админки + логин;
   - ссылку на пользовательскую документацию;
   - контакт техподдержки.
10. Запустить **Onboarding checklist** — кнопка в карточке организации,
    автоматически проверит готовность.

### 16.2. Перевести организацию в другую сеть

1. **Organizations** → открыть карточку → **«Move to another network»**.
2. Выбрать целевую сеть.
3. **Reason**: обязательно. Например: «Слияние сетей по запросу
   `network_operator` Иванова, тикет SUP-12345».
4. Подтвердить.
5. После переноса убедиться, что:
   - бухгалтерия пересчитала биллинг;
   - `network_operator` старой сети потерял доступ;
   - `network_operator` новой сети получил доступ;
   - webhook'и, привязанные к организации, продолжают работать.

### 16.3. Расследовать спорный платёж

1. Меню → **Reports** → **Sales** → найти заказ по номеру / e-mail /
   последним 4 цифрам карты.
2. В карточке заказа: статус, payments, refunds, audit-log, webhooks.
3. Если статус `paid`, но клиент утверждает иное:
   - проверить webhook от провайдера в **Webhooks** → **Deliveries**;
   - открыть LK провайдера (Stripe Dashboard / YooKassa и т.д.) по
     `provider_payment_id` из карточки;
   - сравнить суммы и timestamps.
4. Если найдено расхождение — запустить ручную реконсиляцию
   (раздел 12.2) и зафиксировать в `audit_log` с reason.
5. Если требуется возврат — кнопка **«Initiate refund»** в карточке
   заказа.

### 16.4. Открыть продажу нового мероприятия

1. Убедиться, что у организации есть активная **площадка** и
   **payment config** в режиме `live`.
2. **Events** → **+ New event** (раздел 8).
3. Настроить **категории билетов и цены**.
4. Опубликовать.
5. Подключить или активировать **каналы продаж** (раздел 9).
6. (Опционально) настроить промо-коды и динамические скидки.
7. Уведомить организатора, что мероприятие в продаже.

## 17. FAQ и troubleshooting

### Не приходит письмо со ссылкой для сброса пароля
- Проверьте папку «Спам».
- В **Audit Log** ищите событие `user.password_reset_requested`.
- Если событие есть, но письма нет — проверьте статус mail-провайдера
  в **System → Health**.
- В крайнем случае — сгенерируйте one-time-code (раздел 6.4).

### `403 Forbidden` при cross-tenant действии
- Убедитесь, что у вас есть роль `platform_superadmin` (а не
  `platform_operator`).
- Проверьте, что вы указали `X-Admin-Reason` (если используете API
  напрямую).
- Проверьте `audit_log` — возможно, ваша учётка временно заблокирована.

### Webhook не доходит до клиента
- **Webhooks → Deliveries**: посмотреть последний failed attempt.
- Если код 4xx — проблема на стороне получателя (неверный URL, секрет,
  schema).
- Если код 5xx или timeout — попросите клиента посмотреть свои логи.
- Кнопка **«Replay»** для повтора.

### Несоответствие в реконсиляции
- Открыть **Reconciliation → Discrepancies** → посмотреть raw payload.
- Сравнить с LK провайдера.
- Возможные причины: задержка webhook'а (подождать 15 мин и
  перепроверить), refund вне Arena (потребуется ручная сверка),
  баг интеграции.

### Сканер не пропускает валидный билет
- Проверить, что устройство привязано к нужному event/venue.
- Проверить версию приложения сканировщика.
- Проверить, что время устройства синхронизировано (NTP).
- В **Scanners → Live** посмотреть точный код отказа.

### Коды HTTP-ответов API

| Код | Значение | Что делать |
|---|---|---|
| 200/201 | OK / Created | — |
| 400 | Validation | Поправить payload, см. `error.details.field`. |
| 401 | Unauthorized | Истёк токен — перезайти. |
| 403 | Forbidden | Нет прав / нужен `reason`. |
| 404 | Not Found | Объект не существует или вне scope. |
| 409 | Conflict | Дубликат / устаревший `If-Match`. |
| 422 | Unprocessable | Бизнес-правило нарушено. |
| 429 | Rate limited | Подождать, см. `Retry-After`. |
| 500 | Internal | Поднять инцидент; см. `correlation_id` в логах. |
| 503 | Service unavailable | Подождать 30–60 сек; зависимость недоступна. |

### Куда обращаться
- Внутренний саппорт платформы: `platform-ops@arena.example.com` (или
  Slack-канал `#arena-platform-ops`).
- Инциденты безопасности: `security@arena.example.com` (PGP-ключ в
  `08_architecture/10_compliance_security_privacy_ru.md`).
- Эскалация: `cto@arena.example.com`.
