# Виджет продажи билетов: UX-исследование и выбор технологий

Обновлено: 2026-07-10
Статус: research note + принципы; кодирование не начато. Владелец
подтвердил: user-friendly интерфейс выбора билетов — критический
приоритет всего проекта.

## 1. Что исследовано

1. Легаси-архивы `05_widgets_and_site_templates/widgets/legacy_widget_archives`:
   - `widget2018`, `widget_09_03_2021`, `widget.v2.222.i18n` — PHP +
     jQuery-эпоха: серверные включения (config.php/index.php),
     раздельные js-модули (basket/tickets/orders/city), локализация
     файлами. Встраивание — копированием папки в корень сайта.
   - `fluwid_kraftnum` (актуальный лицензионный виджет Bil24, v3.8.x) —
     **Flutter Web**: `main.dart.js` ≈ 3,7 MB + CanvasKit WASM;
     рендеринг схем — canvas (технология MCharts, залы 5000+ мест);
     Material 3; самодостаточное приложение в папке `/fluwid/`,
     конфигурация URL-параметрами (`id, cityId, frontendId, token,
     zone, seatMode, schemeStyle, promo, decorations…`).
2. Документация fluwid (bil24.pro/fluwid_docs) и инструкции по схемам
   (см. `09_autoforge/seating_backlog.md` §6 — конвенции уже
   зафиксированы там).
3. **Живой прогон покупки** в тестовой зоне (театр Hybernia, событие
   VIVALDIANNO, схема ~1000 мест): полный путь от карточки события до
   Stripe Checkout (sandbox), с подтверждением e-mail кодом и выбором
   двух мест на схеме.

## 2. Разбор UX текущего виджета (fluwid) по шагам прогона

Что сделано хорошо (берём в наш виджет):

- Ценовые категории — цветные чипы над схемой; мгновенно читаемая
  легенда (100/200/300/500 EUR).
- Схема зала целиком, зум-кнопки + fit, номера рядов с обеих сторон,
  сцена подписана. Занятые места визуально «пустые» кружки.
- Плавающая мини-корзина при выборе («2 шт. · 1000 EUR · Перейти в
  корзину») — итог всегда на экране, не надо скроллить.
- Корзина: **видимый обратный отсчёт холда (20:00)**, построчно
  сектор/ряд/место/цена, сервисный сбор отдельной строкой, удаление
  позиций.
- Оплата — редирект на Stripe Checkout с предзаполненным e-mail:
  минимум PCI-скоупа (соответствует нашему guardrail).

Слабые места (наши возможности отстроиться):

1. **E-mail-гейт до всего**: код из письма требуется ДО просмотра
   схемы. Максимальный барьер в самой верхней точке воронки. У нас:
   схема и выбор мест — без идентификации; e-mail — на шаге оплаты.
2. **Тяжёлая первая загрузка**: ~4+ MB JS/WASM до первого рендера.
   Для виджета, встраиваемого в чужие сайты, это боль на мобильных
   сетях. Цель нашего виджета: < 150 KB gzip до интерактивной схемы
   зала ≤ 1500 мест.
3. **Canvas-рендер = ноль доступности**: ни одно место не является
   элементом DOM — скринридеры, клавиатура, автотесты, SEO не работают.
   Наша compliance-цель WCAG 2.2 AA (10_compliance…_ru.md) на canvas
   недостижима.
4. **Тупик после истечения холда**: Stripe-сессия и резерв истекают —
   пользователь получает «You're all done here» без пути назад
   (проверено на живом прогоне). Нужен recovery-флоу: «места
   освободились — забрать те же места заново одним кликом, если ещё
   свободны».
5. Нет подсказки «лучшие доступные места», нет фильтра «показать
   только категорию X», выбор соседних мест — вручную.
6. Кастомизация — только палитра; шрифты/скругления/бренд — нет.
7. **Страница успеха пустая** (проверено полным прогоном до оплаты
   4242-картой): «The payment is completed. If the payment was
   successful, the tickets will be sent…» — англ. текст при русской
   локали, неуверенная формулировка («если платёж прошёл»), ни номера
   заказа, ни купленных мест, ни билетов/QR на экране — только кнопка
   в личный кабинет. Наша страница успеха: номер заказа, список мест,
   QR/PDF сразу на экране + кнопки Wallet, и только потом «также
   отправлено на почту».
8. Плюс, подтверждённый прогоном: истёкший холд корректно возвращает
   места в продажу — повторный заход захватил те же места 18/19
   (важный инвариант, наш SEAT-C1 обязан вести себя так же).

## 3. Рекомендация по технологии

### Вариант A (мост, скорость): переиспользовать fluwid через шлюз

У нас есть лицензионная копия fluwid и Bil24-совместимый шлюз
(`hbil24`). После волн SEAT-D1..D3 (реальные `GET_SEAT_LIST` /
`GET_SCHEMA` / `RESERVATION seatList` + BSS-экспорт) fluwid можно
направить на наш шлюз и получить работающий виджет с рассадкой «в
день запуска». Использовать как временное решение для Palác Akropolis
и как интеграционный тест шлюза. Все слабости §2 при этом наследуются
— это мост, не цель.

### Вариант B (цель): собственный виджет `apps/widget`

- **Язык/фреймворк: TypeScript + Svelte (компилируемый, ~10-30 KB
  рантайм) с упаковкой в Web Component (custom element
  `<arena-tickets>`)**. Альтернатива с теми же свойствами — Lit.
  Причины: минимальный вес, отсутствие рантайм-зависимостей на
  странице-хозяине, Shadow DOM изолирует стили от чужого сайта,
  один тег для встраивания.
- **Рендер схемы: SVG в DOM из канонической geometry JSON** (тот же
  контракт, что SEAT-B3 `GET /v1/event-sessions/{id}/schema`).
  Каждое место — элемент с `role`, `aria-label`
  («Партер, ряд 5, место 18, 500 EUR, свободно»), фокусируемый с
  клавиатуры. До ~3–5 тыс. мест SVG DOM держит 60fps при
  правильной структуре (слои секторов, `transform` для зума,
  делегирование событий). Canvas-режим — отдельный milestone для
  стадионов (large_venue_mode из 04_large_venue…_ru.md), API уже
  спроектирован под это (битсеты статусов).
- **Статусы мест**: снапшот + дельта-поллинг `?since_version=` (уже в
  SEAT-B3); интервал 2–5 с на экране выбора; SSE/WebSocket — резерв.
- **Встраивание**: `<script src=".../widget.js">` +
  `<arena-tickets session-id="…" locale="cs">` ; iframe-fallback для
  строгих CSP; шорткод в нашем WordPress-плагине (apps/wp-plugin).
- Флаттер отклонён для нашего виджета: вес, a11y, кастомизация,
  найм/сопровождение (весь фронтенд проекта — TypeScript).

Решение: делаем B как целевой путь; A включаем только если запуск
Akropolis потребуется раньше готовности B.

## 4. UX-принципы нашего виджета (фиксируем как требования)

1. Схема и цены видны сразу, без регистрации/кода. Идентификация —
   на шаге оплаты (e-mail для доставки билетов), код — только если
   этого требует «Личный кабинет», не покупка.
2. Мобильный жест-набор: pinch-zoom, drag-pan, тап по месту; нижний
   sheet-виджет корзины (Material-паттерн из fluwid оставляем).
3. Место = кнопка: фокус, `aria-label`, клавиатурная навигация по
   рядам стрелками. WCAG 2.2 AA.
4. Таймер холда виден всегда; за 2 мин до истечения — предупреждение;
   после истечения — one-click повторный захват тех же мест (если
   свободны) вместо тупика.
5. «Лучшие доступные»: кнопка авто-подбора N соседних мест в заданной
   категории (сортировка по приоритету рядов из геометрии).
6. Фильтр по категории/цене: клик по чипу цены приглушает остальные
   зоны схемы.
7. Соседние места: при выборе 2+ билетов подсказывать смежные пары.
   Предупреждать о «одиночном месте между выбранными» (single-seat
   gap) — классика театральных правил, включаемая опцией организатора.
8. Сервисный сбор показывать до корзины (в тултипе места/чипе цены) —
   принцип all-in pricing из compliance-дока.
9. Полная темизация токенами (цвета, радиусы, шрифт) + `lng` — как у
   fluwid, но глубже; RTL для he.
10. Телеметрия воронки: schema_viewed → seat_selected → cart →
    payment_started → paid; событийная модель совместима с нашим
    webhook-каталогом (15_webhook_event_catalog.md).

## 5. Что добавить в архитектуру/бэклог (волна SEAT-W, после SEAT-A..E)

1. `apps/widget` — Svelte+TS web component, сборка в один файл,
   бюджет ≤150 KB gzip; демо-страница в repo.
2. Публичный checkout-контракт для виджета: анонимная резервация
   (уже SEAT-C1) + создание checkout-сессии + redirect на провайдера +
   страница результата с deep-link обратно в виджет (recovery-флоу
   п.4 §4 — новый endpoint «повторный захват мест истёкшей
   резервации»).
3. Правило single-seat-gap и приоритеты рядов — расширения geometry
   JSON (`row.rank`, опция организатора `gap_policy`) — заложить в
   схему заранее, продавать не обязательно сразу.
4. Embed-дистрибуция: инструкция + сниппет; интеграция в
   apps/wp-plugin (шорткод/блок Gutenberg).
5. Нагрузочный бюджет: schema+status для зала 1500 мест ≤ 200 KB
   суммарно (gzip), TTFI на 3G ≤ 3 с — включить в load-test сценарии
   (ops/loadtest).
6. Транзитный вариант: конфигурация fluwid на наш `hbil24`-шлюз
   (документировать параметры `frontendId/token` ↔ наши feed-токены)
   — после SEAT-D3.
7. Абстракция канала доставки билетов (§7): тип канала в
   delivery-контракте, Telegram Bot API-адаптер в
   platform/delivery, вариант виджета как Telegram Mini App
   (identity из initData, доставка в чат без e-mail).
8. UI-опции шага оплаты: подсказки опечаток e-mail-доменов,
   переотправка билета со страницы успеха, отключаемое поле телефона
   (флаг канала продаж).

## 6. Решения владельца (2026-07-10) — вопросы закрыты

- **Q-W1 РЕШЕНО:** личный кабинет покупателя — в следующей версии
  виджета (v2). v1: доставка билетов + страница заказа по ссылке.
- **Q-W2 РЕШЕНО:** никаких кодов подтверждения e-mail в покупке
  (исключены ещё на этапе архитектуры/ТЗ). E-mail вводится **ровно
  один раз — на шаге оплаты** (Stripe всё равно его требует), и это
  единственная точка ввода. Обязательна защита от опечаток (§7).
- **Q-W3 РЕШЕНО:** в виджете оплата — через Stripe (redirect,
  Checkout). На WordPress/WooCommerce-сайтах допускается
  использование платёжных плагинов WordPress — это путь
  apps/wp-plugin, не виджета.

## 7. Доставка билетов и контактные данные (решения владельца)

1. **Каналы доставки — абстракция, не «только e-mail».** Если
   пользователь пришёл из мессенджера, где он уже авторизован
   (Telegram-канал/бот, другие мессенджеры), билеты доставляются
   туда **без запроса e-mail вообще**. Реализационное следствие:
   - виджет получает «контекст доставки» при инициализации (embed в
     Telegram Mini App даёт identity пользователя из initData —
     авторизация происходит в момент захода);
   - платформенный слой доставки (delivery_jobs) получает тип канала:
     `email | telegram | <messenger>` — новый adapter под Telegram Bot
     API; шаблоны билетов уже локализованы, PDF/QR переиспользуются.
   - при мессенджер-доставке e-mail для Stripe можно заполнять
     технически (или запрашивать только для чека) — уточним при
     реализации против требований Stripe.
2. **Защита от опечаток в e-mail** (билет, ушедший на несуществующий
   адрес — потерянный клиент): валидация формата + подсказки
   опечаток доменов (gmial.com → gmail.com и т.п.), явное
   отображение адреса на странице успеха с кнопкой «изменить и
   выслать повторно» (повторная отправка билета — уже есть в
   платформе, admin resend).
3. **Конфигурируемые поля покупателя (флаги канала продаж/события).**
   Организаторы хотят собирать данные — но каждое лишнее поле роняет
   мобильную конверсию, поэтому все поля кроме e-mail —
   отключаемые флаги, по умолчанию выключены:
   - **имя** (флаг; печатается на билете как holder name — поле уже
     есть в модели билета);
   - **телефон** (флаг; резервный канал связи на случай ошибочного
     e-mail);
   - задел на будущее: произвольные доп. поля события (анкета) — не
     в v1, но контракт конфигурации полей проектировать списком, а
     не парой булевых.
