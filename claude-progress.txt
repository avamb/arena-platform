Session: 2026-06-22 -- Feature #13 (405 handler returns error envelope with Allow header)
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #13 requires that an existing route hit with the wrong HTTP method
returns 405 with:
  1. The Allow header listing supported methods
  2. Body: standard JSON error envelope with code='http.method_not_allowed'
  3. Content-Type: application/json

KEY DISCOVERY: chi v5 behavior
Chi v5.1.0 does NOT set the Allow header when a custom MethodNotAllowed handler
is registered. The default handler sets Allow via `w.Header().Add("Allow", ...)`,
but when a custom handler is installed, chi returns it directly without the
Allow-header-setting code path.

SOLUTION: chi RouteContext.Routes is an exported field that implements the
chi.Routes interface (includes Match(rctx, method, path) bool). The custom
handler probes all candidate HTTP methods (GET, HEAD, POST, PUT, PATCH, DELETE,
OPTIONS) by calling Routes.Match on a fresh RouteContext. Matched methods are
joined into the Allow header value.

Session: 2026-06-22 -- Feature #92 (PostgreSQL Connection Pool pgx/v5)
"PostgreSQL connection pool (pgx/v5)"
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #92 delivers a clean pgx/v5 pool adapter in a new package
apps/backend/internal/adapters/postgres/.

Three new files:

1. apps/backend/internal/adapters/postgres/pool.go (NEW)

   a. NewPool(ctx, cfg) (*pgxpool.Pool, error)
      Constructs a *pgxpool.Pool using cfg fields:
        MinConns  = cfg.DBPoolMinConns
        MaxConns  = cfg.DBPoolMaxConns
        MaxConnLifetime  = cfg.DBPoolMaxConnLife
        MaxConnIdleTime  = cfg.DBPoolMaxConnIdle
      Issues a 5-second ping to verify connectivity; closes pool and
      returns wrapped error on failure. Returns the raw *pgxpool.Pool
      (no wrapper), unlike platform/database.Open which adds a health-
      checker goroutine.

   b. RegisterPoolMetrics(ctx, pool, m)
      Starts a background goroutine (exits on ctx.Done) that snapshots
      pool.Stat() every 15 seconds and populates all five labels of the
      DBPoolConnections GaugeVec in *observability.Metrics:
        acquired, idle, max, total, new_total
      Also publishes immediately on first call (no initial 15s wait).
      Nil-guards for pool==nil, m==nil, m.DBPoolConnections==nil.

   c. PingProbe struct (implements httpserver.ReadinessProbe)
      NewPingProbe(pool, probeName) *PingProbe
        - probeName defaults to "database" if empty
        - ProbeName() returns the stable /readyz checks map key
        - Ping(ctx) applies a 2-second timeout then calls pool.Ping
          Returns nil on success or wrapped error on failure

   d. TxBeginner interface
      interface { BeginTx(ctx, opts) (pgx.Tx, error) }
      Satisfied by *pgxpool.Pool; allows Tx helper to be tested without
      a real pool.

   e. Tx(ctx, pool, fn) error
      Wraps fn in an explicit pgx transaction (default isolation):
        - Commits if fn returns nil
        - Rolls back (and returns fnErr) if fn returns error
        - Rolls back and re-panics if fn panics
        - Wraps BeginTx and Commit errors with "postgres: begin/commit tx:"

Session: 2026-06-22 -- Feature #12 (404 handler returns error envelope)
"Requests for unknown paths must return the standard error envelope"
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #12 verifies that all unknown paths return a structured JSON error
envelope instead of chi's default plain-text "404 page not found\n" response.
The envelope must contain: code="http.not_found", message, request_id (non-empty),
trace_id (non-empty), and the X-Request-Id response header must be set.

Two changes:

1. apps/backend/internal/platform/httpserver/server.go (UPDATED)

   a. Added handleNotFound() -- a package-level function that calls:
        writeJSON(w, http.StatusNotFound,
            errorEnvelope("http.not_found",
                "the requested resource does not exist", r))
      It reuses the existing errorEnvelope() from devtoken.go (same package)
      which reads request_id from chimw.GetReqID(r.Context()) and trace_id
      from logging.TraceID(r.Context()).

   b. Added s.router.NotFound(handleNotFound) in mountOperationalRoutes().
      This is called AFTER s.mountOperationalRoutes() has already added real
      routes (/healthz, /readyz), so chi builds mx.handler (the full
      middleware chain + routing table). This is crucial — see KEY INSIGHT below.

2. apps/backend/internal/platform/httpserver/not_found_test.go (NEW)

   In-package test (package httpserver) with 6 test functions:
     TestNotFound_StatusIs404                 -- step 1
     TestNotFound_ContentTypeIsJSON           -- step 2
     TestNotFound_BodyMatchesErrorEnvelope    -- step 3 (code, message, request_id, trace_id)
     TestNotFound_RequestIDHeaderIsSet        -- step 4
     TestNotFound_NotChiDefault               -- step 5
     TestNotFound_RequestIDMatchesBetweenHeaderAndBody -- correlation guard

   All 6 tests pass as verified via Docker (`golang:1.24` image).

KEY INSIGHT: chi v5.1.0 NotFound + middleware chain
-----------------------------------------------------
When no routes are registered on a chi.Mux, `mx.handler` is nil. Chi's
ServeHTTP short-circuits to `mx.NotFoundHandler().ServeHTTP(w, r)` DIRECTLY,
bypassing the entire middleware chain (no X-Request-Id, no trace_id set).

Fix: `buildNotFoundRouter()` in the test registers a `/probe` route so chi
builds mx.handler and the full middleware chain wraps ALL requests, including
those handled by the NotFound handler.

In production (Server.New), this is never an issue because mountOperationalRoutes()
registers /healthz and /readyz before NotFound() is called.

Session: 2026-06-22 -- Feature #27 (Worker Graceful Shutdown)
"arena-worker graceful shutdown completes claimed jobs"
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #27 verifies that sending SIGTERM to arena-worker lets currently
claimed jobs finish, stops claiming new jobs, then exits with code 0.

Changes to apps/backend/internal/platform/worker/worker.go:

1. Run() -- changed defer to a combined log+close:
   Previously: defer close(w.doneCh)
   Now:        defer func() {
                   w.logger.Info("shutdown complete")
                   close(w.doneCh)
               }()
   This guarantees Stop() only returns nil AFTER "shutdown complete" is logged.

2. Run() -- added per-job shutdown watcher goroutine:
   Before w.execute(ctx, job), a goroutine selects on:
     - w.stopCh (Stop() called)
     - ctx.Done() (SIGTERM via signal.NotifyContext)
     - jobDone (job finished normally)
   If stop fires while job is still running, it logs:
     "shutdown initiated, finishing 1 claimed job" (job_id, job_type attrs)
   jobDone is always closed after execute() returns -- no goroutine leak.

3. Stop() -- removed "worker stopped cleanly" log (superseded by Run defer).

4. execute() -- context.Background() for MarkDone:
   SIGTERM cancels rootCtx in production; cancelled ctx makes PGQueue.MarkDone
   fail immediately, leaving the job 'claimed'. context.Background() with the
   queue's own 5-10s internal timeout is correct.

5. markFailureOrRetry() -- same context.Background() fix for MarkRetry/MarkFailed.

New file: apps/backend/internal/platform/worker/worker_graceful_shutdown_test.go

Compile-time guard: var _ slog.Handler = (*logCapture)(nil)

4 tests covering all 9 feature steps:

KEY DESIGN DECISIONS
--------------------
  - Fault injection is a boolean field on Server (not a global) so tests can
    create independent fault-on and fault-off server instances in parallel
  - The error code "internal.transaction_failed" is distinct from "audit_failed"
    or "outbox_failed" to reflect the transaction-level failure, not write failure
  - atomicityTx.Begin() panics (step 10) -- the handler never calls it, proven
    by the fact all tests pass without triggering the panic
  - In unit tests, audit goes through captureAuditWriter (in-memory) rather than
    tx.Exec(), so the test verifies audit.WriteTx was called AND tx not committed
  - FAULT_INJECT_OUTBOX_AFTER_AUDIT is deliberately NOT in the validated Config
    struct (to avoid it accidentally being required in production validation)
