Session: 2026-06-22 -- Feature #45 (Concurrent identical Idempotency-Key requests)
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #45 requires that two parallel POST /v1/echo requests with the same
Idempotency-Key both:
  - Return HTTP 200
  - Return byte-identical response bodies
  - Produce at most 1 audit_events row
  - Produce at most 1 outbox_events row
  - Produce exactly 1 idempotency_keys row
  - Never deadlock or 500

ANALYSIS
--------
The previous idempotency Middleware had no concurrency protection: both parallel
requests could get a MISS from Lookup, run the handler independently (producing
different request_ids / echo_event_ids in the body), then the second Save would
be a no-op (ON CONFLICT DO NOTHING). This left identical bodies unguaranteed.

SOLUTION: golang.org/x/sync/singleflight
-----------------------------------------
Added `var group singleflight.Group` inside the Middleware closure (per-route).
In the MISS path, wrapped the handler invocation in `group.Do(sfKey, fn)`:
  - Winner goroutine: runs the handler, captures response, saves to store
  - Waiter goroutines: block in group.Do, share the captured result

KEY BUG DISCOVERED: singleflight.Do sets `shared=true` for ALL callers when
duplicates exist — including the winner. Using `shared` directly caused the winner
to ALSO call replayStored, resulting in double responses.

FIX: Track `wasWinner` bool set inside the fn closure (only executed by winner):
  - winner: wasWinner=true → falls through (response written by capturingWriter)
  - waiter: wasWinner=false → calls replayStored to write replay to their w

CHANGES
-------

1. apps/backend/internal/platform/idempotency/idempotency.go (MODIFIED)
   - Added import golang.org/x/sync/singleflight

   - Replaced MISS path with group.Do wrapping; wasWinner flag for winner/waiter

2. go.mod (MODIFIED)

Session: 2026-06-22 -- Feature #29 (Invalid JSON body returns 400 envelope)
"Invalid JSON body returns 400 envelope"
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Feature #29 requires that POST /v1/echo returns structured 400 responses for
three distinct bad-body scenarios, each with a dotted-namespace error code.

IMPLEMENTATION
--------------

1. apps/backend/internal/platform/httpserver/echo.go
   - Added "errors" to imports
   - Replaced the single json.Unmarshal block with three distinct cases:

   CASE 1 — Empty body (len=0):
     * Returns 400 with code="http.empty_body"
     * Message: "request body is required and must be a JSON object"
     * Checked BEFORE json.Unmarshal to avoid a confusing "unexpected end of
       JSON input" message for this extremely common client mistake

   CASE 2 — Malformed JSON (*json.SyntaxError):
     * Returns 400 with code="http.invalid_json"
     * Message includes byte offset: "parse error near byte offset N"
     * err.Error() alone doesn't include offset; we use errors.As to get the
       *json.SyntaxError and format Offset explicitly
     * Non-syntax decode errors (invalid UTF-8, etc) fall through to
       "request body is not valid JSON: " + err.Error()
     * Raw body NEVER appears in the message (XSS / log injection defence)

   CASE 3 — Wrong JSON shape (*json.UnmarshalTypeError):
     * Returns 400 with code="http.invalid_shape"
     * Message: "expected JSON object but received <type>" (e.g. "array")
     * Detected via errors.As(err, &typeErr) BEFORE the syntax branch
     * Also fixes code="missing_field" → "http.missing_field" for consistency

2. apps/backend/internal/platform/httpserver/echo_invalid_json_test.go (NEW)
   Package: httpserver (in-package, reuses buildEchoServer/mintJWT from
   echo_audit_test.go)
   16 test functions covering all feature steps:

   Step 1 (malformed → 400, code=http.invalid_json):
     TestInvalidJSON_MalformedBody_Returns400
     TestInvalidJSON_MalformedBody_CodeIsHttpInvalidJSON
     TestInvalidJSON_MissingCloseBrace_Returns400

   Step 2 (message references parse error position):
     TestInvalidJSON_MessageReferencesPosition
     TestInvalidJSON_MessageContainsOffset_IsNumeric

   Step 3 (body NOT echoed back):
     TestInvalidJSON_BodyNotEchoedBack           (XSS script tag check)

KEY DISCOVERY: chi v5 behavior
Chi v5.1.0 does NOT set the Allow header when a custom MethodNotAllowed handler
is registered. The default handler sets Allow via `w.Header().Add("Allow", ...)`,

joined into the Allow header value.

      - The 50ms floor ensures the window is reliably larger than any test's

Session: 2026-06-22 -- Feature #57 (POST /v1/echo with empty message returns 400)
"Server validates that 'message' is non-empty. Empty string yields 400 with

WHAT WAS DONE
-------------
Feature #57 requires fine-grained validation of the 'message' field in POST /v1/echo:
  Step 1: {"message":""} -> HTTP 400, code='validation.field_empty'
  Step 2: details.field='message' in the response
  Step 3: {} (missing key) -> 400, code='validation.field_required'
  Step 4: {"message":null} (explicit null) -> 400, code='validation.field_required'

ANALYSIS
--------
The existing echo.go had a single validation at line 131-134:
  if strings.TrimSpace(req.Message) == "" {
    writeJSON(..., errorEnvelope("http.missing_field", ...))

This didn't distinguish between missing/null vs empty string, and didn't include

JSON decode:

  - {"message":null} -> req.Message == ""
  - {"message":""} -> req.Message == ""

To distinguish them, a secondary decode into map[string]json.RawMessage was added.

     * If message present but TrimSpace == "" -> validation.field_empty
     * Both use errorEnvelopeWithDetails with {"field": "message"}
