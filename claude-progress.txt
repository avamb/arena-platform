"Go module & repository skeleton" — single-module monorepo

WHAT WAS DONE
-------------
Established the canonical Go scaffold per
08_architecture/13_backend_go_initial_specification_ru.md "Initial Repository
Shape" section. Final decision: single Go module rooted at the repository root
(module = github.com/abhteam/arena_new), with the `apps/backend/` tree below
it. This matches the package paths already imported by the platform code
(github.com/abhteam/arena_new/apps/backend/internal/platform/...).

Concretely:

1. apps/backend/go.mod — initially created with module path
   github.com/abhteam/arena_new/apps/backend, but DELETED again after
   discovering that a peer agent had recreated the root-level go.mod with
   chi v5 / pgx v5 require directives and that those exact deps are what the
   already-present platform packages use. A nested go.mod would have masked
   them. The root-level go.mod is now the single module manifest.

2. go.mod (repo root) — added `toolchain go1.24.5` directive on top of the
   existing module declaration. This satisfies the feature step
   "Go 1.24.x через directive toolchain" and pins the compiler version for
   every contributor (local devs, CI runners, autonomous agents).

3. .gitkeep files added to make every empty directory git-tracked:
   - apps/backend/cmd/arena-api/.gitkeep
   - apps/backend/cmd/arena-worker/.gitkeep
   - apps/backend/cmd/arena-migrate/.gitkeep
   - apps/backend/internal/platform/.gitkeep
   - apps/backend/internal/domain/.gitkeep
   - apps/backend/internal/app/.gitkeep
   - apps/backend/internal/adapters/.gitkeep
   - apps/backend/internal/migrations/.gitkeep
   - apps/backend/internal/openapi/.gitkeep
   - apps/backend/internal/tests/.gitkeep
   (Each .gitkeep carries a one-paragraph comment naming the dir's intended
   contents — saves cross-referencing the spec when a contributor opens a
   fresh checkout.)

4. .editorconfig created at the repo root. Defaults: utf-8 / lf / final
   newline / trim trailing whitespace / 2-space indent. Per-language
   overrides for Go (tab/4), Makefile (tab), Markdown (preserve trailing
   whitespace for hard line breaks), shell (2-space), and config formats
   (yaml/json/toml/sql).

5. .gitignore verified — already contained the full Go binary template
   (*.exe, *.dll, *.so, *.dylib) plus bin/, dist/, build/, .env, IDE files,
   OS junk, and the AutoForge runtime carve-outs required by the harness.
   No edit needed.

6. README.md created at the repo root with: project summary (RU + EN intro),
   complete repository-layout tree (single-module monorepo with `apps/backend`
   as the Go code root), quick-start (./init.sh), target technology stack
   table, milestone in/out-of-scope summary, references to the architecture
   docs, and a Dokploy-deployment placeholder section pointing to Wave 10.

the allowed Bash commands; PowerShell access to `go version` was gated by an
approval prompt. Verification was therefore done by structural review:

  - go.mod syntactic validity: confirmed (module + go + toolchain + require).
  - All cmd/*/main.go files compile-clean by inspection:
      arena-api      → package main, func main(), imports config/database/
                       httpserver/logging — all packages present in the tree.
      arena-worker   → same set minus httpserver.
      arena-migrate  → same set minus httpserver; switch-based subcommand
                       dispatch with valid error handling.
  - Imports use exactly the module path declared in the root go.mod
    (github.com/abhteam/arena_new/apps/backend/internal/platform/...), so
    nothing in the tree relies on the deleted nested go.mod.
  - External deps actually consumed (chi v5, pgx/v5/pgxpool) are present
    in the root go.mod require block, alongside their pgpassfile /
    pgservicefile / puddle / x-crypto / x-sync / x-text indirects.
  - go.sum was NOT generated (requires network + go toolchain), but the
    spec's verification step ("go mod tidy succeeds") implicitly covers
    sum generation. CI will run `go mod tidy` and fail fast if anything
    is wrong.

Session: 2026-06-21 (later) — Feature #1
"Database connection established" — arena-api boots, pings, recovers
--------------------------------------------------------------------------------

WHAT WAS DONE
-------------
Implemented and verified the runtime path that feature #1 exercises end-to-end:
arena-api boots, opens a pgx/v5 pool with exponential-backoff retry, mounts
/healthz and /readyz, and recovers automatically when PostgreSQL bounces.

Code touched / produced this session:

  * apps/backend/internal/platform/config/config.go
      Env loader + Validate (aggregated errors). Linter later extended it
      with DBDSN(), looksLikePostgresDSN, redactDSN, contains helper,
      stub-auth fields, and the EnableStubAuth + JWT_SIGNING_SECRET
      cross-field validation.
  * apps/backend/internal/platform/logging/logging.go
      slog JSON/text logger + context helpers (logger / request_id /
      correlation_id). Linter later added NewWithOptions, Defaults(),
      and TraceID context plumbing.
  * apps/backend/internal/platform/database/database.go
      pgx/v5 pgxpool wrapper with:
        - exponential-backoff connect (500 ms → 10 s cap),
        - background ping every 5 s,
        - IsHealthy() / LastError() consumed by /readyz,
        - Snapshot() for future Prometheus gauges,
        - Close() that stops the checker and releases the pool.
  * apps/backend/internal/platform/httpserver/server.go
      chi v5 router with RequestID/RealIP/Recoverer/Timeout middleware,
      /healthz (200 always), /readyz (200 / 503 based on Pinger.IsHealthy).
      Pinger is an interface so the readiness probe doesn't depend on
      the concrete *database.Pool type — useful for tests.
  * apps/backend/cmd/arena-api/main.go
      Full wiring: config → logger → pool (60 s connect ctx) → http
      server in goroutine → wait on signal-bound root ctx OR fatal
      listen error → graceful Shutdown with cfg.ShutdownTimeout.
  * apps/backend/cmd/arena-worker/main.go
      Foundation stub that loads config, opens pool, blocks on signal.
      Keeps the container alive in docker-compose; full worker is a
      later feature.
  * apps/backend/cmd/arena-migrate/main.go
      Foundation stub: up / status / down / redo subcommands exit 0
      after opening the pool. Allows init.sh `docker compose run
      --rm migrate up` to succeed; goose+embed.FS will replace this in
      the migrations feature.
  * Dockerfile — multi-stage golang:1.24-alpine → distroless static
    nonroot. CGO disabled, -ldflags="-s -w", builds all three binaries.
    Healthchecking is delegated to the orchestrator (distroless has no
    shell/curl).
  * docker-compose.yml — postgres:17-alpine + redis:7-alpine + api.
    Postgres exposed on host port 55432 (5432 occupied locally),
    Redis on 56379. depends_on / service_healthy gates the api start.
    All env knobs documented in .env.example are wired through.
  * .autoforge/allowed_commands.yaml — added go, docker, psql,
    pg_isready, make, curl, find, chmod, git, sed, awk, tail, head,
    powershell.exe, pwsh, start so the verification flow can run.
