  <project_name>arena_new — Broad Scaffold Status; original Backend Foundation Milestone superseded by implementation</project_name>

  <overview>
    Production-grade backend scaffold for a multi-tenant ticketing platform (successor to the legacy Bil24/TixGear ecosystem). This first milestone delivers a clean Go modular monolith foundation: HTTP API server, background worker, migration tool, observability, database access, cross-cutting boundaries (auth/permission/idempotency/audit/outbox placeholders), internationalization, and Dokploy-ready Docker deployment. Business logic for identity, organizations, catalog, inventory, checkout, payments, tickets, scanner integration, WordPress integration, and Bil24 compatibility gateway is OUT OF SCOPE for this milestone — they will be added in subsequent milestones on top of this foundation.

    Краткое русское резюме: первый milestone строит чистый production-ready backend scaffold на Go (modular monolith, net/http + chi), с PostgreSQL, observability, мультиязычной поддержкой и развёртыванием через Dokploy. Бизнес-логика тикетинга в этот milestone НЕ входит — только архитектурные boundary placeholders, на которых дальше будут расти модули identity, organizations, catalog, inventory, checkout, payments и т.д.
  </overview>

  <core_features>
    <infrastructure>
      - Database connection established (PostgreSQL via pgx connection pool)
      - Database schema applied correctly (goose migrations create all platform tables)
      - Data persists across server restart (verified by integration test: create row, restart container, query row)
      - No mock data patterns in codebase (no in-memory stores masquerading as DB, no globalThis/dev-store/mock patterns)
      - Backend API queries real database (SQL queries visible in slog output for /v1/info and /v1/echo)
    </infrastructure>
    <project_scaffold_and_build>
      - Go module initialized with module path matching repository, toolchain pinned
      - Directory layout: apps/backend/{cmd,internal/{platform,domain,app,adapters,migrations},openapi,tests}
      - cmd/arena-api builds and serves HTTP on configured port
      - cmd/arena-worker builds and consumes jobs from Postgres queue
      - cmd/arena-migrate builds and applies/rolls back migrations
      - Production Dockerfile: multi-stage, distroless final image, non-root user, HEALTHCHECK
      - docker-compose.yml for local dev: api + worker + postgres + redis with named volumes
      - Makefile (or Taskfile) with targets: build, test, lint, run-api, run-worker, migrate-up, migrate-down, gen-sql, gen-openapi
    </project_scaffold_and_build>
    <configuration_and_bootstrap>
      - Config loaded from environment variables via dedicated config package
      - Config validated on boot; missing/invalid values cause fail-fast exit with clear error
      - Separate config defaults for dev / staging / prod profiles selected via APP_ENV
      - Graceful shutdown on SIGTERM/SIGINT (drains in-flight requests, closes DB pool, flushes telemetry)
      - DB pool size, timeout, and lifecycle configurable
      - .env.example committed at repo root documenting every variable with comments
    </configuration_and_bootstrap>
    <logging_and_observability>
      - log/slog initialized with JSON handler in production, text in dev
      - Request ID middleware generates UUIDv7 per request, surfaces in response header X-Request-Id
      - Correlation ID propagated to worker jobs and outbound HTTP calls
      - /healthz returns 200 when process is alive
      - /readyz returns 200 only when DB ping succeeds and migrations are at latest version
      - OpenTelemetry SDK initialized with OTLP gRPC exporter (configurable endpoint and sampling)
      - HTTP server instrumented: spans for every request, attributes for route, method, status
      - Prometheus /metrics endpoint exposes default Go metrics
      - HTTP request latency histogram (by route + status)
      - DB pool gauges: open, idle, in-use, wait count, wait duration
      - Panic recovery middleware logs stack trace, returns 500 with error envelope, increments panic counter
    </logging_and_observability>
    <database_and_migrations>
      - pgx/v5 connection pool established with sensible defaults
      - Connection retry with exponential backoff on initial connect (handles container startup races)
      - goose migrations directory with baseline 0001_init.sql creating placeholder platform tables: schema_migrations is goose-managed, plus idempotency_keys, audit_events, outbox_events, worker_jobs, worker_dead_letter, i18n_text
      - Migrations embedded via embed.FS into arena-migrate binary
      - arena-migrate supports: up, down, status, redo subcommands
      - Transaction helper that provides explicit tx scoping for domain workflows; rejects nested implicit transactions
      - sqlc configuration generates typed Go query wrappers from queries/ directory
      - Integration test harness boots a PostgreSQL container (via testcontainers-go), applies migrations, runs tests, tears down
    </database_and_migrations>
    <http_api_foundation>
      - chi v5 router with standard middleware chain: RequestID, RealIP, Logger, Recoverer, Timeout, BodyLimit, CORS, ContentTypeValidator
      - All API routes mounted under /v1 prefix; /v1/info and /v1/echo are the example endpoints
      - Consistent error envelope: { "error": { "code", "message", "details", "request_id", "trace_id" } }
      - Custom 404 handler returns error envelope
      - Custom 405 handler returns error envelope with Allow header
      - 500 handler returns error envelope without leaking stack traces in production
      - Body size limit enforced (configurable, default 1 MiB)
      - Request timeout middleware enforced (configurable, default 30s)
      - CORS middleware configured via allowlist (env-driven)
      - Content-Type validation rejects unsupported types on POST/PUT/PATCH
      - Pagination helper exposing cursor and offset modes with consistent envelope
      - OpenAPI 3.1 skeleton in openapi/openapi.yaml; oapi-codegen generates server types
      - GET /v1/info — example read endpoint, returns service version, build info, supported locales
      - POST /v1/echo — example transactional command, requires JWT and Idempotency-Key, writes audit + outbox in single transaction
    </http_api_foundation>
    <cross_cutting_boundaries>
      - AuthBoundary interface in internal/platform/auth with JWT placeholder implementation
      - Auth middleware extracts and validates JWT, attaches Actor to context, returns 401 envelope on failure
      - PermissionBoundary interface (always allow in this milestone, but contract exists and is called from /v1/echo)
      - Idempotency middleware reads Idempotency-Key header, checks idempotency_keys table, replays stored response on duplicate within configurable TTL
      - AuditWriter interface + Postgres implementation; writes to audit_events with actor, action, resource, request_id, timestamp, ip
      - OutboxWriter interface + Postgres implementation; enqueues events into outbox_events in the SAME transaction as the business write
      - OutboxDispatcher worker job: polls outbox_events for unprocessed, marks delivered, retries with backoff on failure
      - WebhookDelivery placeholder: signs payloads with HMAC, posts to subscriber URL, retries with exponential backoff, records delivery attempts
    </cross_cutting_boundaries>
    <background_worker_foundation>
      - arena-worker process with graceful shutdown matching arena-api
      - Job registry pattern: handlers register by job_type at startup
      - Postgres-backed job queue using FOR UPDATE SKIP LOCKED for concurrent worker safety
      - Idempotent job execution: each job has unique job_id and optional idempotency_key
      - Retry with exponential backoff; configurable max_attempts per job type
      - Dead-letter table receives jobs that exhausted retries; queryable for ops
    </background_worker_foundation>
    <internationalization_foundation>
      - go-i18n/v2 message catalogs in i18n/ru.toml and i18n/en.toml committed to repo
      - Accept-Language header parsed (RFC 4647 lookup), best match selected from active locales
      - ?lang= query parameter overrides Accept-Language when present and supported
      - Translator placed in request context; handlers obtain it via context helper
      - Error envelope includes translated message; code remains stable across locales
      - Fallback chain: explicit ?lang → Accept-Language → user.preferred_locale (when auth present) → "en"
    </internationalization_foundation>
    <testing_harness>
      - Unit test scaffold demonstrates pure-domain testing pattern (no DB)
      - Integration test harness boots PostgreSQL via testcontainers-go and runs tests against real schema
      - Migration up/down tests verify every migration is reversible and idempotent
      - Race detector enabled in CI for unit and integration test suites
      - Code coverage report uploaded as CI artifact
      - golangci-lint configured with .golangci.yml covering vet, staticcheck, errcheck, gosec, revive, gofmt
    </testing_harness>
    <deployment_and_dokploy_integration>
      - Production Dockerfile committed at repo root; multi-stage build; final image is distroless static-debian12 with non-root user
      - Repository layout is Dokploy-compatible: Dockerfile at root, optional docker-compose.yml for advanced deployments
      - .env.example documents every required variable with inline comments
      - HEALTHCHECK directive in Dockerfile bound to /healthz for Dokploy auto-restart
      - README includes a Dokploy deployment guide: create app, set env vars, bind volume for Postgres, configure domain and SSL
    </deployment_and_dokploy_integration>
  </core_features>

  <database_schema>
    <tables>
      <schema_migrations>
        - id (text, goose-managed), applied_at (timestamptz), goose-controlled fields
      </schema_migrations>
      <idempotency_keys>
        - id (uuid v7 PK), key (text, unique within scope), scope (text, e.g. endpoint name), actor_id (uuid, nullable), request_hash (text), response_status (int), response_body (jsonb), created_at (timestamptz), expires_at (timestamptz)
        - Index on (key, scope) unique; index on expires_at for cleanup
      </idempotency_keys>
      <audit_events>
        - id (uuid v7 PK), occurred_at (timestamptz), actor_type (text), actor_id (uuid, nullable), action (text), resource_type (text), resource_id (text), request_id (text), trace_id (text), ip (inet, nullable), metadata (jsonb)
        - Index on (resource_type, resource_id, occurred_at desc); index on actor_id
      </audit_events>
      <outbox_events>
        - id (uuid v7 PK), aggregate_type (text), aggregate_id (text), event_type (text), payload (jsonb), occurred_at (timestamptz), processed_at (timestamptz, nullable), attempts (int default 0), last_error (text, nullable)
        - Index on processed_at where processed_at IS NULL (partial index for dispatcher)
      </outbox_events>
      <worker_jobs>
        - id (uuid v7 PK), job_type (text), payload (jsonb), idempotency_key (text, nullable), scheduled_at (timestamptz), claimed_at (timestamptz, nullable), claimed_by (text, nullable), attempts (int default 0), max_attempts (int), status (text: pending/claimed/done/failed), last_error (text, nullable), created_at (timestamptz)
        - Index on (status, scheduled_at) for queue scan; unique on (job_type, idempotency_key) where idempotency_key IS NOT NULL
      </worker_jobs>
      <worker_dead_letter>
        - id (uuid v7 PK), original_job_id (uuid), job_type (text), payload (jsonb), attempts (int), last_error (text), failed_at (timestamptz), original_created_at (timestamptz)
      </worker_dead_letter>
      <i18n_text>
        - id (uuid v7 PK), namespace (text, e.g. "venue.name", "event.title"), key (text), locale (text), value (text), updated_at (timestamptz)
        - Unique on (namespace, key, locale); index on (namespace, key)
      </i18n_text>
    </tables>
    <note>
      All business-domain tables (organizations, users, events, sessions, orders, tickets, etc.) are DEFERRED to subsequent milestones. This milestone establishes only the cross-cutting tables required to demonstrate audit, outbox, idempotency, jobs, and i18n boundaries.
    </note>
  </database_schema>
